## 🔍 调研内容

### 1. ELECTRE-I 方法回顾

**经典 ELECTRE-I** (精确数版本):

**核心概念**:
- **和谐指数 (Concordance Index)**: 衡量方案 A 在某些准则上优于方案 B 的程度
- **不和谐指数 (Discordance Index)**: 衡量方案 A 在某些准则上劣于方案 B 的程度
- **级别优于关系 (Outranking Relation)**: 基于和谐/不和谐阈值判定 A 是否优于 B

**算法步骤**:
1. 计算和谐矩阵 C = [c_ij]
2. 计算不和谐矩阵 D = [d_ij]
3. 确定和谐阈值 c̄ 和不和谐阈值 d̄
4. 提取核 (Kernel): A ≻ B 当且仅当 c_ij ≥ c̄ 且 d_ij ≤ d̄

### 3. 技术可行性分析

#### 3.1 区间和谐指数计算

**挑战**:
- 如何定义"区间 A 在准则 j 上优于区间 B"?
- 可能度法: P(A_ij ≥ B_ij) > 0.5
- 但和谐指数是权重求和，如何融合可能度?

**可能的方案**:
```
c_ij = Σ (w_j * P(A_ij ≥ B_ij))
```

**问题**:
- 公式复杂，计算量大
- 语义不清晰（可能度的加权和是什么意思？）
- 缺乏学术验证

#### 3.2 区间不和谐指数计算

**挑战**:
- 经典公式: d_ij = max_{j: B_j > A_j} [(B_j - A_j) / (max_j - min_j)]
- 区间版本: 如何定义"区间 B_j > 区间 A_j"?
- 使用可能度? 还是区间宽度?

**可能的方案**:
```
d_ij = max_{j: P(B_j ≥ A_j) > 0.5} [width(B_j - A_j) / (max_j - min_j)]
```

**问题**:
- 公式不统一
- 计算复杂
- 缺乏学术支持

#### 3.3 区间核提取

**挑战**:
- 经典方法: 提取级别优于关系图的核
- 区间版本: 关系图变得复杂（可能度不是二元关系）
- 如何定义传递性?
- 如何提取核?

**可能的方案**:
- 使用可能度阈值（如 P(A ≥ B) ≥ 0.7）
- 构建有向图
- 提取强连通分量的核

**问题**:
- 阈值如何确定?
- 算法复杂度高
- 缺乏实现参考

## ⚠️ 调研结论

### 学术支持评估

| 评估维度 | 评分 | 说明 |
|---------|------|------|
| 论文数量 | ⭐ (1/5) | < 10 篇，非常稀少 |
| 引用次数 | ⭐ (1/5) | 大多数 < 20 次 |
| 算法明确性 | ⭐⭐ (2/5) | 理论探讨多，具体算法少 |
| 实现案例 | ⭐ (1/5) | 几乎没有开源实现 |
| 技术可行性 | ⭐⭐ (2/5) | 复杂度高，风险大 |
| **总体评分** | **⭐⭐ (2/5)** | **不推荐实现** |

### 主要发现

1. **学术支持严重不足**:
   - 过去10年相关论文 < 10 篇
   - 绝大多数研究聚焦 ELECTRE-III, ELECTRE-TRI
   - ELECTRE-I 区间版本是**非主流方向**

2. **技术复杂度高**:
   - 区间和谐指数计算公式不统一
   - 区间不和谐指数定义困难
   - 区间核提取算法复杂
   - 缺乏参考实现

3. **用户需求低**:
   - ELECTRE-I 本身使用率低于 TOPSIS, TODIM, VIKOR
   - 区间版本需求更低
   - 投入产出比不合理

4. **风险高**:
   - 实现难度大（预计 5-6 人日）
   - 算法正确性难以验证
   - 性能可能较差（O(n³) 复杂度）

## 🎯 建议决策

### 决策: 延迟到 v1.0+

❌ **不建议在 v0.9 实现 ELECTRE-I 区间版本**

**理由**:
1. ✅ 学术支持不足（⭐⭐/5）
2. ✅ 技术复杂度高（5-6人日）
3. ✅ 用户需求低（ELECTRE-I 本身使用率低）
4. ✅ 风险高（正确性难以验证）
5. ✅ 投入产出比不合理

### 替代方案

**方案 A**: 聚焦核心价值
- ✅ 优先完成 VIKOR + TODIM 区间版本
- ✅ 将节省的工期用于提升质量和文档
- ✅ 降低技术风险

**方案 B**: 作为可选功能
- ⏳ 延迟到 v1.0+
- ⏳ 如果用户需求强烈，再考虑实现
- ⏳ 预留算法设计文档

**推荐**: **方案 A**
